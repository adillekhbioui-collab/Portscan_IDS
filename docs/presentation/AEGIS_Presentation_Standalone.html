The Threat: Port Scanning

Anas E.

Reconnaissance Phase

Port scanning is the first step of nearly all targeted cyberattacks. Attackers use it to:

  • Map open ports and services
  • Fingerprint the Operating System
  • Identify unpatched vulnerabilities
Stealthy scans (low-and-slow) evade traditional detection by blending into background noise.

Limitations of Traditional IDS

  • Static Rules: Tools like Snort and Suricata rely on rigid thresholds.
  • Evasion: Attackers easily bypass rules by lowering scan rates or using fragmented packets.
  • Alert Fatigue: SOC analysts receive thousands of false positive alerts daily from legitimate monitoring tools.

Objectives & Methodology

Success Criteria

To be viable for production, AEGIS must achieve:

  • F1-Score: ≥ 88%
  • Accuracy: ≥ 90%
  • False Positive Rate (FPR): < 6%

Combined with an Active Defense response (MTD + Honeypots).

CRISP-DM Framework

  • Business Understanding: Reduce SOC fatigue.
  • Data Understanding: CICIDS2017 Dataset.
  • Data Preparation: IQR Capping, Scaling, SMOTE.
  • Modeling: RF, XGBoost, Isolation Forest.
  • Evaluation: Strict Anti-Leakage constraints.
  • Deployment: Active bridging and Live Dashboard.

Dataset Overview

Canadian Institute for Cybersecurity Intrusion Detection System.

The Majority Anomaly Paradox

Adil

Why did Isolation Forest fail?

Isolation Forest assumes that anomalies are rare (usually < 5%).

In our CICIDS2017 subset, PortScans represent 55.48% of the dataset.

The Paradox: The algorithm isolated the BENIGN traffic faster than the PortScan traffic, essentially classifying normal traffic as anomalies.

Production Remediation

To deploy IF effectively, it must be trained in a semi-supervised manner exclusively on pure BENIGN traffic (~100,000 flows) before inference.

System Architecture

        ▼

        │
        ▼

        │
        ▼

        │
        ├─ BENIGN → [Log silently]
        │
        └─ PORT_SCAN → [Deception / Defense] + [Dashboard Alert]

Severity Mapping

El Yazid

AEGIS maps the AI's confidence score to an autonomous defense action:

Confidence Score Severity Level Autonomous Action Taken
< 0.85 LOW Honeypot redirection only (Monitor)
0.85 – 0.95 MEDIUM Honeypot + IP Blacklist (Drop via UFW)
> 0.95 HIGH Honeypot + Blacklist + Port Mutation (MTD)
The Dashboard consumes these events via Socket.IO APIs (`/api/alert`, `/api/block`, `/api/honeypot_event`) to alert the SOC in real-time.

Active Defense Subsystem

El Yazid

Modules

  • Core Deception: 5 honeypot profiles feeding `shadow_node_interaction=1`.
  • Network Mutator (MTD): Dynamically shifts active service ports via iptables.
  • Traffic Shaper: SYN-ACK tarpit and MSS sabotage to stall scanners.
  • IP Manager: TTL-based auto-unblock and kernel blackholing.

Empirical Results

  • +420% time wasted by attackers (scanning becomes 5x slower).
  • 85% → 20% drop in attacker OS-fingerprinting success.
  • +2.3% gain in ML accuracy in live environments via the honeypot feedback loop.
